## Go-to-Market: Atlanta Ecosystem

Atlanta has 100+ entrepreneurial centers with a dense, recurring cohort model:
- Engage (2/year), gener8tor (8 companies/program), ATDC (multiple stages)
- Atlanta Tech Village (pre-accelerator), Women's Entrepreneurship Initiative (annual)
- Fintech South (10 companies), 30+ corporate innovation centers

**Why Atlanta is the right first market:**
- Enough operators within 30 miles to iterate without travel/sales cost
- All run the same basic workflow: cohort intake → curriculum → network → next stage
- Can validate without needing national scale

**Best first operator target:**
ATDC or Women's Entrepreneurship Initiative — both have a coordinator model, recurring cohorts, and a "what happens after the cohort ends" problem the platform solves directly.

**What we're selling operators:**
Not community software. A product that reduces coordinator overhead, keeps cohort members connected post-program, and makes the program look more sophisticated to funders via pathway tracking.

## Business Model: Multi-Sided Revenue

Five revenue streams identified:

| Stream | Mechanism | MVP approach |
|---|---|---|
| Members | Ticket purchase (course, program, membership) | Kit Commerce → enrollment |
| Mentors | Embedded in ecosystem at right pathway stage, lead pipeline | Manual invite as co-admins for now |
| Sponsors | Company + companion setup, program alignment | Manual one-time setup |
| Incubators | Operator fee — they pay directly, admin sets up their program | Manual, you are the operator |
| Community grants | Platform delivers grant-funded programs, nonprofit fiscal sponsor | External contract, report outcomes |

**Key tension identified:** Platform must not compete with mentors/coaches who are members.
**Resolution:** Platform handles orientation and peer learning. Mentors handle execution and accountability. Platform creates qualified demand for mentoring — a prepared member is a better coaching client.

## Deployment Unit Question

Four levels considered: cohort (<10), meetup (<50), incubator (<200), community (200+).

**Conclusion:**
- Cohort level only works if YOU are the facilitator (personal use case, direct revenue)
- Incubator level (50–200) is where institutional budget and coordinator champion exist
- **Run your own cohorts first** — generates revenue, forces features to work, creates a live demo for operator sales
- Meetup level is a lead generation play, not a revenue play

**The model that threads it:**
Position as the operating system for cohort facilitators. That persona (coach, fractional executive, program manager) has direct revenue tied to the platform working well and brings it up to their institution.

## Feature Architecture: What Exists

### Correct terminology (clarified today)
- **Programs** = cohort containers (the program IS the cohort)
- **Sessions** = events that happen during program execution (meetings, workshops, standups)
- Cohorts is legacy/deprecated terminology

### Programs ↔ Offerings relationship
`market_offerings.linked_program_id` already exists — an offering is the marketplace face of a program.

**Full connected chain (already built):**
```
Market (discovery bucket)
  └── Offering (pricing, Kit Commerce, CTA)
        └── linked_program_id → Program (cohort container)
              ├── Circle (community discussion)
              ├── Journeys + content (structured learning)
              ├── Sessions (scheduled events)
              └── Participants (state machine: invited → enrolled → active → completed)
```

**Platform admin** (`/platform-admin/offerings`) is where program/offering linking, pricing, Kit Commerce product ID, and enrollment behavior are configured. Operator-controlled, not self-serve.

**Company Companion** — curated resource panel on a company profile. Pins pathways, documents, checklists so market visitors know what they're getting into before purchasing.

## Monetization Model: Ticket System

### Core principle
No annual subscriptions — too much refund risk at early stage. One-time ticket purchases instead.

### Ticket taxonomy

| Ticket type | What it grants |
|---|---|
| `membership` | user_class upgrade, circle allowance included |
| `class_upgrade` | user_class bump (more circles, more features) |
| `season` | Access to all programs in a cohort cycle |
| `program` | Enrollment in one specific program |
| `course` | Enrollment in one course |
| `offering` | Access to a specific market offering |

### Ticket lifecycle
```
Listed (inventory available, not yet on sale)
  → Purchased (Kit Commerce payment complete)
    → Pending (valid_from date not yet reached)
    → Waitlisted (sold out, in queue)
  → Redeemed (access granted, enrollment activated)
  → [Expired — future, not implemented now]
```

### Season tickets
A season maps to a cohort cycle (Spring 2026, Fall 2026). A season ticket grants access to all programs running in that cycle. Admin controls when tickets go on sale (`status: upcoming → on_sale`) and when they become redeemable (`on_sale → active`).

### Bundle tickets
Pre-packaged sets of tickets. Cafeteria model — member's bundle may include N program enrollments or a class upgrade. Bundle purchase auto-generates the constituent access grants.

### Circle access model
Circles are NOT individually ticketed. Circle access is an entitlement of user_class:
- user_class 3 = up to N circles
- user_class 4 = up to M circles
- Upgrade prompt shown when at limit, not hard block
- Leave a circle → slot freed immediately

**MVP decision:** Set circle allowance high (10+) for all classes at launch. Do not enforce limits until circles have active content and participation worth paying for.

## MVP Monetization Priority

**Revenue at early stage will come from courses and programs, not membership or circles.**

The only monetization chain that needs to work for MVP:
```
Member finds course or program in market
  → purchases via Kit Commerce
    → access ticket created
      → participant enrolled in program
        → member accesses circle, sessions, journeys
```

Everything else (season tickets, bundles, class upgrades, circle enforcement) is Phase 2 once there's evidence of what people actually want to buy.

**Open question:** Has the Kit Commerce → access ticket → enrollment chain been tested end-to-end with a real purchase? This is the gate before inviting paying members.
